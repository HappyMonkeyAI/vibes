🏗️ Architecture

Vibes is a TypeScript application running on Node.js, using Ink 6 + React 19 for the terminal UI and the OpenAI SDK for LLM communication. The architecture follows a layered pattern: a React TUI layer communicates with an agent orchestration layer that manages LLM calls, tool execution, and persistence.

Agent Orchestration Layer

Mission Planner (src/agent/mission-planner.ts)

Receives a user's natural language prompt and decomposes it into a structured plan: milestones → tasks with dependencies. Uses a dedicated planner model (or the main model if PLANNER_MODEL is empty). Performs tech-stack detection (one I/O scan of package.json + file tree) at plan time and attaches it to the mission.

DAG Scheduler (src/agent/scheduler.ts)

Manages task state with a directed acyclic graph. Tracks dependencies, runs ready tasks in parallel up to MAX_CONCURRENT_TASKS, discovers new tasks during execution, and handles human-in-the-loop interventions via promise-blocking. Interventions are resolved via a direct ref (not React state) to keep the agent loop "warm."

Task Executor (src/agent/task-executor.ts)

Per-task agent loop. For each task:

  1. Builds system prompt with Codex RAG context (if enabled), mission context, tech stack, and available tools
  2. Sends prompt to LLM, streams response
  3. Parses tool calls (native API or markdown fallback for models that can't render tool schemas)
  4. Executes tools, collects results, feeds back into the loop
  5. Detects thrash (consecutive identical tool sequences)
  6. Compresses messages when approaching token limit (hard cap at 150 messages)
  7. Runs structural audit on completion (stack-aware code quality checks)

Context Manager (src/agent/context-manager.ts)

Token counting via gpt-tokenizer with LRU cache (512 entries). Message compression with optional force flag. Estimates token budget and truncates or compresses history to fit.

Triage Agent (src/agent/triage-agent.ts)

Optional pre-flight classifier that categorizes incoming tasks by domain, complexity, and risk. Can auto-steer the mission plan based on classification results. Configured via TRIAGE_ENABLED and TRIAGE_INTERVAL.

Agent Loop Flow

User prompt
    │
    ▼
Mission Planner ─── detects tech stack ─── produces milestones + tasks
    │
    ▼
DAG Scheduler ─── resolves deps, dispatches ready tasks
    │
    ├── Task Executor (slot 1) ─── LLM loop ─── tools ─── review ─── audit
    ├── Task Executor (slot 2) ─── LLM loop ─── tools ─── review ─── audit
    │
    ▼
Completion ─── LTM update ─── next mission

Memory is stored at .antigravity/memories/ in three categories:

  • Semantic — codebase_insights/, architectural_decisions/
  • Procedural — patterns_and_lessons.md
  • Episodic — history/ (plan/walkthrough archives, handoffs)

Config System

Configuration flows through a Zod schema (src/config.ts) with 37 fields. Sources merge in this order:

Zod defaults ← .env ← .vibes/config.json ← runtime UI edits

The config object is a singleton mutated in-place via Object.assign by updateConfig(). Modules import config directly and read fields at call-time (never destructured at module level to avoid stale references).

Safety Systems

  • Auto-git snapshots — pre-mission git commit for undo capability
  • Thrash detection — consecutive identical turn tool sequences trigger pause
  • YOLO mode — bypasses safety limits when user explicitly enables it
  • Message hard cap (150) — prevents OOM in long-running agent loops
  • Codex circuit breaker — 5-min backoff on Neo4j connection failure
  • Structural audit — stack-aware code quality checks after each task
