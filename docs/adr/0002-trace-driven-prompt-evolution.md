# ADR 0002: Trace-Driven Prompt Evolution (AEGIS-Inspired)

- Status: Proposed
- Date: 2026-06-16
- Task: Integration of HarnessX (2606.14249) concepts

## Context

The performance of the `vibes` agent relies heavily on the quality and specificity of its prompt harness and tool definitions (e.g., `gemma-12b-prompt.md` and `model-prompts.ts`). Currently, these harnesses are hand-crafted and static. If the agent repeatedly fails a task due to a misunderstanding of a tool or a local convention, it has no built-in mechanism to dynamically adapt its own core instructions based on those failures.

Recent research ([HarnessX: A Composable, Adaptive, and Evolvable Agent Harness Foundry - arXiv:2606.14249](https://arxiv.org/abs/2606.14249)) demonstrates that evolving runtime interfaces using execution traces (the "AEGIS" engine) can yield significant performance gains (+14.5% on average).

Vibes already records execution trajectories passively via `src/agent/trace.ts` into `.vibes/traces/<taskId>.jsonl`. We need an active loop that distills these traces into permanent harness improvements.

## Decision

We will introduce a **Trace-Driven Prompt Evolution** mechanism to the `vibes` architecture.

The architecture will introduce the following components:

1. **Evolution Manager (`src/agent/evolution-manager.ts`)**
   - A post-mission or triage-triggered process that parses `.vibes/traces/<taskId>.jsonl`.
   - It will identify execution antipatterns (e.g., repeated tool failures, ignored system instructions, inefficient loops).
   - Based on identified antipatterns, it will generate targeted "behavioral corrections".

2. **Dynamic Prompt Foundry (`src/agent/model-prompts.ts` refactoring)**
   - Move away from monolithic, static fallback strings.
   - Implement a composable prompt builder.
   - The builder will accept the base `gemma-12b-prompt.md`, but dynamically inject the corrections generated by the Evolution Manager.
   - This forms a "Substitution Algebra" where specific prompt blocks can be swapped or augmented based on historical performance.

3. **Persistent Lessons Memory**
   - The generated corrections must be saved durably so that future sessions benefit from past mistakes.
   - We will write these derived rules into `.antigravity/memories/patterns_and_lessons.md` or a newly designated `evolution_rules.md` file, which is loaded into the context at session start.

## Consequences

### Positive
- **Self-Healing:** The agent will gradually stop repeating the same mistakes across different runs.
- **Model Agnostic Adaptation:** This improves performance without requiring costly fine-tuning, leveraging the context window for behavior adaptation.
- **Stronger Local Execution:** Tailors the generic fallback prompts strictly to the idiosyncrasies of the user's specific workspace and toolset.

### Negative
- **Context Bloat Risk:** If the Evolution Manager is too aggressive, it could append too many rules, consuming valuable token budget. The evolution engine must be capable of pruning obsolete rules.
- **Complexity:** Introduces another layer of abstraction between the static files and the prompt sent to the LLM.

## Implementation Steps

1. Create `src/agent/evolution-manager.ts` capable of parsing JSONL traces and passing them to an LLM for "lesson extraction".
2. Refactor `src/agent/model-prompts.ts` to fetch these dynamically generated lessons and inject them into the `executor` role contract.
3. Hook the Evolution Manager into the end of `src/agent/mission-planner.ts` or as a background task.
